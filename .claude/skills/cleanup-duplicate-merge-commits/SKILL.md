# Cleanup duplicated commits in merge history

This skill squashes commits with duplicated subject lines on top of a known
base commit (typically the latest "merge upstream/main into main" commit on a
downstream OCP rebase branch), folding each duplicate into the earliest
occurrence of the same subject. The end goal is a smaller, cleaner linear
history whose final tree is **byte-identical** to the pre-cleanup HEAD.

## When to use

Trigger phrases:

- "cleanup merge history from duplicated commits"
- "squash the duplicate carry commits"
- "deduplicate rebasebot commits"
- "fixup duplicate commit subjects"

Typical setup: a detached HEAD (or branch) on top of a merge commit, with
many `UPSTREAM: <carry>:` commits where several share the same subject
(notably `UPSTREAM: <carry>: rebasebot: update OCP manifests and fix golang
deps`, which often appears 10+ times).

## Non-negotiable invariants

1. **Zero code changes.** `git diff <BACKUP_REF> HEAD` must be empty at the
   end. Equivalently, `git rev-parse <BACKUP_REF>^{tree}` must equal
   `git rev-parse HEAD^{tree}`.
2. **No branch refs are moved.** Operate on the detached HEAD. Leave it to
   the user to update branch pointers / push.
3. **A backup ref exists** before any history-rewriting command runs.
4. **No `--force` pushes.** This skill never pushes.

If invariant 1 cannot be satisfied at the end, ask the user — do NOT silently
accept tree drift.

## Required input

Ask the user (or take from context) for the **base commit** — the commit
that the cleanup should rebase **onto** (exclusive). This is typically the
"merge upstream/main into main" commit. Everything between this commit and
HEAD will be considered for deduplication.

## Strategy: A (keep earliest) — default and recommended

For each duplicated subject:

- The **earliest** occurrence remains as `pick`.
- All later occurrences are moved to immediately follow it, changed to
  `fixup` (messages discarded; subjects are identical anyway).

This avoids reordering "real" commits and minimizes conflicts. Confirm
with the user if they prefer Strategy B (keep latest) — that is rarely
worth the conflict pain.

## Step-by-step procedure

### 1. Survey

```bash
# Confirm detached state and current HEAD.
git status
git rev-parse HEAD

# Count commits and list duplicate subjects above the base.
BASE=<base-commit-sha>
git log --oneline "$BASE"..HEAD | wc -l
git log --format='%s' "$BASE"..HEAD | sort | uniq -c | sort -rn | awk '$1>1'
```

Report the duplicate distribution to the user before proceeding.

### 2. Create a safety backup ref

```bash
git branch backup/pre-squash HEAD
git rev-parse backup/pre-squash    # remember this SHA
```

If `backup/pre-squash` already exists from a prior attempt, pick a fresh
name (e.g. `backup/pre-squash-2`).

### 3. Run the reordering rebase

Use the bundled `reorder_todo.py` script as `GIT_SEQUENCE_EDITOR`. It
rewrites the rebase todo to keep the first occurrence of each subject as
`pick` and converts later duplicates to `fixup` placed immediately after.

Apply `-X theirs` so rebasebot-style "full regeneration" duplicates merge
cleanly without manual conflict resolution. This is safe **only because**
we verify tree-equivalence in step 4; any drift gets corrected there.

```bash
SCRIPT="$(git rev-parse --show-toplevel)/.claude/skills/cleanup-duplicate-merge-commits/scripts/reorder_todo.py"
GIT_SEQUENCE_EDITOR="python3 $SCRIPT" \
    git rebase -i -X theirs "$BASE"
```

If the rebase stops on a conflict that `-X theirs` did not resolve, do
**not** make arbitrary choices. Pick the side that matches the
corresponding file's content at `backup/pre-squash` (or, if both sides are
clearly subsets of that content, pick whichever, since step 4 will catch
drift). Then `git add` and `git rebase --continue`.

### 4. Verify tree equivalence (the critical check)

```bash
git diff backup/pre-squash HEAD
[ "$(git rev-parse backup/pre-squash^{tree})" = "$(git rev-parse HEAD^{tree})" ] \
    && echo OK || echo TREES_DIFFER
git log --oneline "$BASE"..HEAD | wc -l
git log --format='%s' "$BASE"..HEAD | sort | uniq -d   # must be empty
```

The diff **must** be empty. If it is empty, proceed to step 6.

### 5. Repair tree drift (only if step 4 shows differences)

`-X theirs` may pick the wrong side when the same file is regenerated by
both a duplicate (e.g. a rebasebot commit) and an interleaved "real"
commit (e.g. "Bump OCP for 4.22"). The resulting drift is usually a few
lines (e.g. a `createdAt:` timestamp in a CSV manifest).

Repair procedure — for each drifted file:

```bash
# Identify which commits in the NEW history touch the file.
git log --oneline "$BASE"..HEAD -- <drifted/file/path>
```

Pick the **latest** commit in the new history that legitimately owns
changes to that file (e.g. the latest "Bump OCP" or "Update manifests"
commit), and amend it via autosquash:

```bash
# Restore the correct content from the backup.
git checkout backup/pre-squash -- <drifted/file/path>

# Create a fixup targeting the chosen owner commit.
git commit --fixup=<owner-sha> -s

# Autosquash it in.
GIT_SEQUENCE_EDITOR=true git rebase -i --autosquash "$BASE"
```

Re-run step 4. Repeat until the diff is empty.

If the drift is in a file that has no obvious "owner" commit in the new
history, amend HEAD instead (`git commit --fixup=HEAD` then autosquash, or
just `git commit --amend --no-edit` after staging). The placement matters
less than the final tree being correct.

### 6. Report and stop

Output for the user:

- Backup ref name and SHA (e.g. `backup/pre-squash → 6c0c0833d`).
- Before/after commit counts above the base.
- The remaining commit list (`git log --oneline "$BASE"..HEAD`).
- Confirmation that `git diff backup/pre-squash HEAD` is empty.
- Reminder that HEAD is still detached and no branch refs were moved.
  The user updates their branch / pushes as they see fit.

Provide the rollback command explicitly: `git reset --hard backup/pre-squash`.

## Things to avoid

- **Do not** start without the backup ref. Without it, a botched rebase
  is recoverable only via `git reflog`, which is fragile under concurrent
  activity.
- **Do not** drop commits (`drop` / removing lines from the todo). The
  goal is squash, not deletion. The final tree must match HEAD exactly.
- **Do not** reorder non-duplicate "real" commits. Only duplicates move,
  and they only move to sit next to their earlier twin.
- **Do not** use `squash` instead of `fixup` for these duplicates. The
  duplicate subjects are identical; combining messages just produces
  noisy commit bodies.
- **Do not** push or move any branch pointer. The user owns that step.
- **Do not** edit files unrelated to fixing tree drift in step 5. If a
  conflict tempts you to "improve" something, stop and ask.

## Reference: real-world numbers from the original run

On the `rebasebot/rebase-bot-main` branch with base
`11ee09d480898763341d72adfc2d6ec74cd84281`:

- Before: 85 commits, 13 distinct duplicated subjects (the
  `rebasebot: update OCP manifests and fix golang deps` subject appeared
  13 times by itself).
- After: 61 commits, zero duplicate subjects, tree byte-identical.
- One drifted `createdAt:` line in
  `manifests/stable/manifests/kubernetes-nmstate-operator.clusterserviceversion.yaml`
  was repaired by autosquashing a fixup onto the `Bump OCP for 4.22`
  commit (step 5).
